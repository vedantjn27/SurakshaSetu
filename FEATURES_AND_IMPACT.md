# SurakshaSetu — Features & Impact

> AI-powered federated UBID platform for secure business identity  
> Karnataka Commerce & Industries | Version 1.0.0

---

## Overview

SurakshaSetu (सुरक्षा सेतु — "Bridge of Security") is a federated identity resolution platform that assigns a **Unique Business Identity (UBID)** to every business entity operating across multiple Karnataka government departments. It eliminates duplicate records, detects dormant or fraudulent entities, and gives analysts a single 360° view of any business — all while keeping raw PII protected through deterministic cryptographic scrambling.

---

## Core Features

### 1. Federated UBID Registry

Every registered business receives a single canonical identifier: `UBID-KA-<PIN>-<SHORTCODE>`, e.g. `UBID-KA-560058-A3F2C1`.

- Records from different departments (Factories, KSPCB, BESCOM, Labour, Shop & Establishment) are automatically linked under one UBID
- UBID clusters maintain a full **merge history** — every link, split, and override is permanently recorded
- Anchoring via PAN or GSTIN provides near-deterministic identity confirmation
- Supports **UBID splitting** when a merged cluster turns out to contain two distinct entities
- Network graph endpoint (`/ubid/{id}/network`) visualises all linked records and relationships

---

### 2. AI-Powered Record Linkage Pipeline

The ingestion pipeline processes raw CSV uploads and resolves identity automatically:

| Stage | What Happens |
|-------|-------------|
| **Normalisation** | Business names are lowercased, ASCII-folded (handles Unicode/Kannada), punctuation stripped, abbreviations expanded (`pvt` → `private`, `engg` → `engineering`), and legal suffixes extracted |
| **Address Parsing** | Addresses are parsed into structured components: plot number, locality, city, PIN code |
| **PAN / GSTIN Validation** | Format regex validation, cross-check that GSTIN prefix matches the business's PAN |
| **Blocking** | Candidate pairs are generated using PIN code + Metaphone phonetic key buckets, avoiding O(n²) comparisons |
| **Feature Extraction** | 8 signals computed per pair: Jaro-Winkler name similarity, token-sort ratio, PIN exact match, locality similarity, plot similarity, PAN exact, GSTIN exact, phone exact |
| **Confidence Scoring** | Logistic regression model (if trained) or weighted sum from `rules.yaml`; PAN/GSTIN exact match enforces a hard floor of 0.88 |
| **Resolution** | Score ≥ 0.85 → auto-link; 0.40–0.85 → human review queue; < 0.40 → kept separate |

All thresholds and weights live in `config/rules.yaml` — no code changes needed to tune the system.

---

### 3. Mistral AI Explanations

Every match decision comes with a plain-English explanation generated by Mistral AI:

- Explains *why* the system linked two records in language suitable for a government analyst
- Operates **exclusively on scrambled (hash) data** — raw PII is never sent to the LLM
- Falls back to template-based explanations if the Mistral API is unavailable
- Powers the **natural language query endpoint** (`POST /query/ask`) for ad-hoc analytics

---

### 4. Activity Classification Engine

Every UBID is continuously classified based on incoming government event data:

| Status | Meaning | Score Threshold |
|--------|---------|----------------|
| **Active** | Business is operating | ≥ 0.35 |
| **Dormant** | Reduced activity, possibly inactive | 0.05 – 0.35 |
| **Closed** | No evidence of operation | ≤ 0.05 |

Event types with configurable weights and exponential half-life decay:

- `utility_reading` — electricity/water consumption (weight: 0.45, half-life: 90 days)
- `licence_renewal` — licence renewed (weight: 0.30, half-life: 365 days)
- `inspection_pass` — satisfactory inspection (weight: 0.20, half-life: 180 days)
- `compliance_filing` — return filed (weight: 0.18, half-life: 120 days)
- `licence_lapsed` — licence expired (negative weight: -0.25)
- `no_consumption` — zero utility consumption (negative weight: -0.20)

---

### 5. PII Scrambling & Privacy Playground

All sensitive data exposed in the UI or sent to AI is **deterministically scrambled**:

- Uses HMAC-SHA256 with a secret seed — same real value always maps to the same hash
- Scrambled format: `hash:<64-hex-chars>` — unreadable but consistently comparable
- Admin-only unscramble endpoint for authorised lookup
- The **Privacy Playground** page lets analysts test scramble/unscramble interactively
- Zero raw PII ever reaches the frontend or any third-party service

---

### 6. Human-in-the-Loop Review Workflow

Ambiguous matches (score 0.40–0.85) enter a structured human review queue:

- Reviewers see both records side-by-side with feature scores and AI explanation
- Available actions: **Approve** (link), **Reject** (keep separate), **Escalate** (senior review)
- Approved decisions feed back as labelled training data for future model improvement
- Orphan events (activity data for unrecognised business IDs) have a separate assignment interface

---

### 7. Role-Based Access Control

Three distinct roles with enforced permissions:

| Role | Permissions |
|------|------------|
| **Admin** | Full access: ingest data, manage users, scramble/unscramble PII, view all audit logs, override UBIDs |
| **Reviewer** | Ingest data, approve/reject review queue items, view UBID registry |
| **Analyst** | Read-only: query, view UBIDs and audit logs, run AI queries |

- JWT authentication (HS256, 8-hour expiry)
- Self-registration is restricted to the Analyst role; Admin/Reviewer accounts require administrator creation
- Route protection enforced at both middleware (Next.js) and API (FastAPI Depends) levels

---

### 8. Multi-Department Data Ingestion

Supports structured CSV ingestion for 5 Karnataka departments out of the box:

| Department | ID Field | Key Columns |
|-----------|---------|------------|
| Shop & Establishment | `reg_number` | business_name, address, pin_code |
| Factories | `reg_number` | business_name, address, pin_code |
| Labour | `reg_number` | business_name, address, pin_code |
| KSPCB | `consent_number` | business_name, address, consent_date |
| BESCOM | `account_number` | business_name, pin_code |

- Ingestion runs as a **background job** — large files don't block the API
- Job status polling endpoint: `GET /ingest/status/{job_id}`
- Full SHA-256 content hashing prevents duplicate ingestion of the same file
- New departments can be added purely through `rules.yaml` — no code changes required

---

### 9. Immutable Audit Trail

Every system decision is permanently logged:

- Data ingestion jobs (records processed, new UBIDs created, duplicates skipped)
- Auto-link decisions with confidence scores
- Human review decisions (who approved/rejected, timestamp)
- PAN/GSTIN validation failures
- UBID splits and merges
- Admin-level PII unscramble operations

Searchable audit log UI with action-based colour coding and full pagination.

## Impact

### Problem Solved

Karnataka's commerce and industry data is fragmented across 5+ independent department silos. The same business may appear as:
- "Peenya Engg Pvt Ltd" in Factories
- "Peenya Engineering Private Limited" in Shop & Establishment
- Account #48291 in BESCOM
- Consent #KA-PCB-2019-3821 in KSPCB

Without a shared identity layer, there is no way to detect that these are the same entity — leading to regulatory blind spots, duplicate compliance burdens, and missed enforcement opportunities.

### What SurakshaSetu Delivers

| Outcome | Mechanism |
|---------|----------|
| **Single source of truth** | One UBID per business across all departments |
| **Reduced manual deduplication** | Auto-link pipeline handles ≥ 85% of matches without human review |
| **Ghost business detection** | Activity classifier identifies dormant/closed entities still holding licences |
| **Fraud risk reduction** | PAN/GSTIN cross-validation catches mismatched registrations |
| **Audit readiness** | Immutable, timestamped log of every identity decision |
| **Privacy compliance** | Raw PII never leaves the backend; all AI operations use scrambled data |
| **Analyst empowerment** | Natural language queries via Mistral AI — no SQL knowledge required |
| **Operational efficiency** | New department data ingested in minutes via CSV drag-and-drop |

### Designed for Scale

- Blocking strategy limits candidate comparisons to O(n × bucket_size) rather than O(n²)
- Async background ingestion pipeline handles large CSV files without API timeouts
- MongoDB Atlas scales horizontally; all hot query fields are indexed
- Logistic regression model can be retrained offline and hot-swapped via `models/matcher_lr.joblib`
- All business rules externalised to YAML — policy changes require zero deployments
